# az-staticWeb-template

GitHub template repository for the **Internal Static Content Hosting Platform** (TSP-269).

This repo is cloned by the [bootstrap automation](https://github.com/techops/az-staticWeb-bootstrap) to create per-site repositories (`az-staticWeb-<siteName>`).

## Repository Structure

```
├── .github/workflows/
│   ├── deploy-site.yml      # Deploys site content to both SWA instances on push
│   └── deploy-infra.yml     # Runs Terraform plan/apply on infra changes
├── infra/
│   ├── main.tf              # SWA resources (primary ue2 + DR uc1)
│   ├── frontdoor.tf         # Front Door endpoint, origin group, origins, route, custom domain, DNS, WAF
│   ├── monitoring.tf        # Application Insights + availability tests + alerts
│   ├── variables.tf         # Variable definitions
│   ├── providers.tf         # AzureRM provider config
│   ├── backend.tf           # Terraform state backend
│   └── terraform.tfvars.json  # Generated by bootstrap — site-specific values
├── site/
│   ├── index.html           # Your site content goes here
│   └── staticwebapp.config.json  # Generated by bootstrap — Entra auth config
└── README.md
```

## How It Works

1. **Bootstrap creates your repo** from this template via a Jira ticket request
2. **Bootstrap generates** `terraform.tfvars.json` and `staticwebapp.config.json` with your site config
3. **RRE reviews** the bootstrap PR and merges it
4. **`deploy-infra.yml`** runs Terraform to create your SWA instances + Front Door config
5. **You push content** to the `site/` folder and it auto-deploys to both regions

## Need Backend Services?

This template is for **static-only sites** by default. If your site needs a Function App, Storage Account, or Key Vault, company has a hardened standard (public + Entra-only + Managed Identity, no VNet) — see [`agents.md` → "Adding a Function App and Storage Account"](agents.md#adding-a-function-app-and-storage-account) for the full Terraform shape and gotchas. Reference implementation: [`techops/az-staticWeb-campaign-planning-tool/infra/`](https://github.com/techops/az-staticWeb-campaign-planning-tool/tree/main/infra) — first new site that exercised the standard from scratch.

Three pre-flight items need CloudOps coordination before your first Function App / Storage apply (policy exemption, runner-MI ABAC expansion, data-role pre-grants). Specifics in `agents.md`.

## Deploying Content

Push your static content to the `site/` directory on `main`:

```bash
git add site/
git commit -m "Update site content"
git push origin main
```

The `deploy-site.yml` workflow will automatically deploy your static HTML files to both SWA regions (East US 2 + Central US) in parallel.

## Required Secrets

These are configured automatically during provisioning:

| Secret | Purpose |
|---|---|
| `SWA_DEPLOY_TOKEN_UE2` | Deployment token for primary SWA (East US 2) |
| `SWA_DEPLOY_TOKEN_UC1` | Deployment token for DR SWA (Central US) |
| `AZURE_CLIENT_ID` | OIDC client ID for Terraform |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Neon subscription ID |

## Architecture

- **Arch Doc:** [TSP-269 Cloud Architecture](https://atlassian.net/wiki/spaces/TECH/pages/6497959970)
- **Platform:** Azure Front Door Premium + Azure Static Web Apps (Standard)
- **Auth:** Entra ID SSO (all sites require authentication)
- **Regions:** East US 2 (primary) + Central US (DR) with automatic failover
- **Monitoring:** Application Insights availability tests from 5 global regions
