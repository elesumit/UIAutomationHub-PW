# Agents.md ‚Äî Veradigm Static Web Site Repository

This file describes everything an AI agent needs to know to work effectively in this repository. Read this entire document before making any changes.

---

## What This Repo Is

This is a **static web site repository** for Veradigm's Internal Static Content Hosting Platform (TSP-269). It was automatically generated by the [az-staticWeb-bootstrap](https://github.com/veradigm-techops/az-staticWeb-bootstrap) automation from a Jira ticket request.

The site is hosted on **Azure Static Web Apps (SWA)** behind **Azure Front Door Premium** with a custom domain under `internal.veradigm.com`. Authentication is handled by **Microsoft Entra ID** (Azure AD) using a shared app registration (`app-staticweb-sso`).

This is a **PaaS-only** architecture. There are no VMs, no containers, no servers to manage. Never introduce IaaS components (VMs, AKS, container instances).

By default the site is static-only. If your team needs backend services (Function App, Storage Account, Key Vault), follow the **¬ß Adding a Function App and Storage Account** section below ‚Äî Veradigm has a standard for how these are wired together (public + Entra-only + Managed Identity, no VNet) and several non-obvious gotchas that will bite you if you deviate.

## How the Platform Works ‚Äî End to End

### The Bootstrap Pipeline (how this repo was created)

1. Someone creates a Jira ticket requesting a new static site (project CLPASD)
2. Jira webhook fires `repository_dispatch` to `az-staticWeb-bootstrap`
3. Bootstrap automation:
   - Reads Jira custom fields (site_name, team_name, team_members, cost_center, business_unit)
   - Creates this repo from the `az-staticWeb-template`
   - Generates `infra/terraform.tfvars.json` with site-specific values
   - Generates `site/staticwebapp.config.json` with Entra auth config and Front Door integration
   - Generates `.github/CODEOWNERS` with team members as `/site/` owners
   - Replaces `SITE_NAME` placeholder in `backend.tf` with the actual site name
   - Copies plain HTML starter content into `site/`
   - Opens a PR assigned to the RRE (aharden) for review

### Infrastructure Deployment (what happens after PR merge)

When the bootstrap PR is merged to `main`, `deploy-infra.yml` runs:

1. **Terraform Validate** ‚Äî fmt, validate, plan (runs on PRs too)
2. **Terraform Apply** ‚Äî creates all Azure resources:
   - 2x Azure Static Web Apps (Standard tier) ‚Äî primary (East US 2) + DR (Central US)
   - Front Door endpoint (`ep-<site_name>`)
   - Front Door origin group (`og-<site_name>`) with both SWAs as origins (priority-based failover)
   - Front Door route connecting the custom domain to the origin group
   - Front Door custom domain with managed TLS certificate
   - DNS CNAME record (`<site_name>.internal.veradigm.com` ‚Üí FD endpoint)
   - DNS TXT record for domain validation (`_dnsauth.<site_name>`)
   - Application Insights (workspace-based) + availability test + metric alert
3. **Post-apply steps** (Azure CLI via OIDC):
   - Adds the custom domain to the shared WAF security policy
   - Extracts SWA deployment tokens and sets them as repo secrets
   - Triggers post-provisioning on the bootstrap repo (Jira update, Slack notification)

### Content Deployment (ongoing)

Any push to `main` that changes files in `site/` triggers `deploy-site.yml`:

1. Copies `site/*` into a `build/` directory
2. Deploys to both SWA instances in parallel (primary + DR)
3. Uses `SWA_DEPLOY_TOKEN_UE2` and `SWA_DEPLOY_TOKEN_UC1` secrets (set automatically by deploy-infra)

### Manual Step After First Deploy

After the initial infrastructure deployment, someone with Entra portal access must manually add the redirect URI to the shared app registration:

- **Azure Portal** ‚Üí App registrations ‚Üí `app-staticweb-sso` ‚Üí Authentication
- Add: `https://<site_name>.internal.veradigm.com/.auth/login/aad/callback`

This is required for Entra ID SSO to work. Without it, users will get an AADSTS50011 error when trying to log in. This step cannot be automated because the managed identity lacks Global Admin permissions for Microsoft Graph API.

## Making Content Updates

Content lives in the `site/` directory. This is the only directory most team members should need to touch.

### How to update site content

1. Create a branch from `main`
2. Edit files in `site/` (HTML, CSS, JS, images, etc.)
3. Open a PR ‚Äî CODEOWNERS will automatically request review from the designated team members
4. Once approved and merged, `deploy-site.yml` automatically deploys to both SWA instances

### Rules for site content

- **Plain HTML only** ‚Äî no build tools, no npm, no frameworks. The deploy workflow copies `site/*` directly to SWA with `skip_app_build: true`. If you add a framework (React, Next.js, etc.), the deploy will still work but the build step won't run, so you'll deploy source code instead of built output.
- **Inline styles and scripts are allowed** ‚Äî the Content-Security-Policy includes `'unsafe-inline'` for both `style-src` and `script-src`.
- **No server-side code in `site/`** ‚Äî `site/` is for static assets only. SWA-managed Functions live in `api/`, and standalone Function Apps are provisioned via Terraform; both require the lockdown pattern in **¬ß Backend Services & Lockdown**.
- **All content is behind Entra authentication** ‚Äî every route (`/*`) requires the `authenticated` role. There are no public pages. Anonymous users are redirected to Entra login automatically.
- **Do not modify `staticwebapp.config.json`** unless you understand the implications. This file controls authentication, routing, security headers, and Front Door integration. Incorrect changes can break authentication or expose the site to direct access bypassing Front Door.

### What `staticwebapp.config.json` does

| Section | Purpose |
|---------|---------|
| `auth.identityProviders.azureActiveDirectory` | Entra ID SSO configuration (shared app registration) |
| `routes` | All routes require `authenticated` role |
| `responseOverrides.401` | Unauthenticated users get 302 redirect to Entra login |
| `forwardingGateway.requiredHeaders` | Locks SWA to only accept traffic from Front Door (X-Azure-FDID) |
| `forwardingGateway.allowedForwardedHosts` | The custom domain that Front Door forwards |
| `globalHeaders` | Security headers (CSP, X-Frame-Options, etc.) |

**Critical**: The `forwardingGateway` section requires **SWA Standard tier** to function. If the SWA is on Free tier, this section is silently ignored and auth redirects will go to the `azurestaticapps.net` default hostname instead of the custom domain. Both SWAs in this repo are provisioned as Standard tier ‚Äî do not downgrade.

## Making Infrastructure Updates

Infrastructure lives in the `infra/` directory. Only the RRE (aharden) can approve infrastructure changes (enforced by CODEOWNERS).

### How to update infrastructure

1. Create a branch from `main`
2. Edit Terraform files in `infra/`
3. Open a PR ‚Äî `deploy-infra.yml` will run `terraform plan` and post the output
4. RRE reviews the plan and approves
5. On merge, `terraform apply` runs automatically

### Rules for infrastructure

- **PaaS only** ‚Äî do not add VMs, AKS clusters, container instances, or any IaaS resources. The platform architecture is deliberately PaaS-only for minimal operational overhead.
- **Do not modify provider configuration** ‚Äî `providers.tf` configures OIDC authentication with a specific managed identity and subscription. Changing these will break Terraform auth.
- **Do not modify backend configuration** ‚Äî `backend.tf` points to a shared Terraform state storage account with a site-specific key. Changing the key will orphan the state.
- **Do not modify `terraform.tfvars.json` manually** ‚Äî this file was generated by the bootstrap automation and contains values derived from the Jira ticket. If values need to change, update them carefully and understand the downstream impact on resource naming.
- **SWA must stay on Standard tier** ‚Äî both `sku_tier` and `sku_size` must be `"Standard"` in `main.tf`. The `sku_size` attribute in AzureRM v4 defaults to `"Free"` and overrides `sku_tier`, so if you remove `sku_size` the SWA will silently downgrade to Free tier. This breaks `forwardingGateway` (auth goes to wrong domain) and disables custom domain support.
- **Do not create per-site WAF security policies** ‚Äî Azure Front Door allows only one security policy per WAF policy per profile. The shared WAF policy (`Ue1NePrdFrontDoorSecPolicy`) is managed in the Neon parent repo. The `deploy-infra.yml` workflow automatically adds this site's custom domain to the shared policy.
- **Front Door managed TLS certificates can take 30-60 minutes to propagate** after initial creation. During this time, browsers may show certificate errors. This is normal Azure behavior, not a configuration issue.

## Authentication Flow

1. User visits `https://<site>.internal.veradigm.com`
2. Front Door routes the request to the primary SWA (or DR if primary is down)
3. SWA checks for authentication cookie ‚Äî if missing, returns 401
4. `staticwebapp.config.json` overrides 401 ‚Üí 302 redirect to `/.auth/login/aad`
5. SWA redirects to Microsoft Entra ID login page
6. User authenticates with corporate credentials (Entra tenant `21d8e422-7fd3-4634-8c8a-01dfde9a5502`)
7. Entra redirects back to `https://<site>.internal.veradigm.com/.auth/login/aad/callback`
8. SWA sets authentication cookie and serves the requested page

**Important**: The `forwardingGateway` section in `staticwebapp.config.json` ensures auth redirects use the custom domain (via Front Door) instead of the default `azurestaticapps.net` hostname. This only works on SWA Standard tier.

## Troubleshooting

### Site shows certificate error (ERR_CERT_COMMON_NAME_INVALID)
- **After initial deploy**: Front Door managed TLS certificates take 30-60 minutes to propagate. Wait and retry.
- **After that**: Check that the Front Door custom domain `deploymentStatus` is `Succeeded` and the endpoint `deploymentStatus` is not stuck at `NotStarted`.

### Site redirects to azurestaticapps.net instead of custom domain
- The SWA is likely on **Free tier**. Check `sku_size` in `main.tf` ‚Äî both `sku_tier` and `sku_size` must be `"Standard"`.
- Or the `forwardingGateway` section is missing/incorrect in `staticwebapp.config.json`.

### AADSTS50011 ‚Äî Reply URL does not match
- The redirect URI `https://<site>.internal.veradigm.com/.auth/login/aad/callback` has not been added to `app-staticweb-sso` in Entra. This is a manual step after first deploy.

### HTML renders as unstyled plain text
- Check the `Content-Security-Policy` header in `staticwebapp.config.json`. It must include `style-src 'self' 'unsafe-inline'; script-src 'self' 'unsafe-inline'` to allow inline styles and scripts.

### Deploy fails with "SWA_DEPLOY_TOKEN not set"
- Infrastructure hasn't been deployed yet. The deploy tokens are set by `deploy-infra.yml` after Terraform apply. Wait for the infra workflow to complete.

### Terraform state lock error
- Another Terraform operation is running (or crashed). Check GitHub Actions for in-progress runs. If a run crashed, the lock may need to be broken manually: `terraform force-unlock <LOCK_ID>`.

## Adding a Function App and Storage Account

Veradigm has a **standard** for how Function Apps and Storage Accounts are wired together, hardened by lessons learned from the `media-curation` app in May 2026. **Follow this pattern from day one ‚Äî don't deviate without a strong reason.** Reference implementation: [`veradigm-techops/az-staticWeb-veradigm-campaign-planning-tool`](https://github.com/veradigm-techops/az-staticWeb-veradigm-campaign-planning-tool/tree/main/infra) ‚Äî the first new site that exercised it from scratch.

### The standard ‚Äî public + Entra-only + MI

The platform direction (Sai Gajavalli, 2026-05) is:

- **Storage accounts are publicly reachable on the internet, but accept Entra ID tokens only.** Shared keys are disabled. Humans access via Entra SSO + tenant Conditional Access MFA. Services access via SystemAssigned Managed Identity.
- **No VNet, no private endpoints, no private DNS zones.** Function Apps are NOT VNet-integrated. Storage and KV sit on their public endpoints, gated by Entra at the credential layer.
- This is the inverse of the older "lock everything behind a VNet" pattern. Don't bring that back ‚Äî `media-curation` migrated AWAY from it and the gotchas are documented below.

### Required Terraform shape ‚Äî Storage Account

```hcl
resource "azurerm_storage_account" "data" {
  name                     = var.storage_account_name
  resource_group_name      = data.azurerm_resource_group.swa.name
  location                 = var.regions.primary
  account_tier             = "Standard"
  account_replication_type = "GRS"           # or LRS for runtime/scratch
  account_kind             = "StorageV2"

  min_tls_version                 = "TLS1_2"
  https_traffic_only_enabled      = true
  allow_nested_items_to_be_public = false

  # NON-NEGOTIABLE: keys off, Entra-only.
  shared_access_key_enabled = false

  # Public endpoint reachable; firewall opens to internet, auth is the gate.
  public_network_access_enabled = true

  network_rules {
    default_action = "Allow"
    bypass         = ["AzureServices"]
  }

  tags = var.tags
}
```

Out-of-band bootstrap (run BEFORE the first `terraform apply`):

```bash
RUNNER_MI=<value from var.mi_client_id>
SA_ID=/subscriptions/.../resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/<sa>
for ROLE in "Storage Blob Data Contributor" "Storage Queue Data Contributor" "Storage Table Data Contributor"; do
  az role assignment create --assignee $RUNNER_MI --role "$ROLE" --scope "$SA_ID"
done
```

Then in `storage.tf`, declare three `azurerm_role_assignment` resources for the runner MI on the SA. Pair them with a `time_sleep` resource (120s) so any data-plane-requiring child resource (containers, the campaigns table, etc.) waits for RBAC propagation before TF tries to create it ‚Äî see the campaign-planning-tool's `storage.tf` for the wiring.

### Required change ‚Äî `providers.tf`

The azurerm provider **must** have `storage_use_azuread = true`. Without it, the provider's own data-plane refresh defaults to Shared Key, fails with `403 KeyBasedAuthenticationNotPermitted`, and you can't `terraform plan` again until you fix it.

```hcl
provider "azurerm" {
  features {}
  resource_provider_registrations = "none"
  use_oidc                        = true
  tenant_id                       = "21d8e422-7fd3-4634-8c8a-01dfde9a5502"
  subscription_id                 = "54305029-7d35-40a9-8bf9-950963b449cc"
  client_id                       = var.mi_client_id
  storage_use_azuread             = true   # ‚Üê REQUIRED with shared_access_key_enabled = false
}
```

If you need to create a Storage Table (see "the giovanni gotcha" below), also add the `azapi` provider:

```hcl
required_providers {
  azurerm = { ... }
  azapi   = {
    source  = "Azure/azapi"
    version = ">= 2.0.0, < 3.0.0"
  }
  time    = {
    source  = "hashicorp/time"
    version = ">= 0.11.0, < 1.0.0"
  }
}
```

### Required Terraform shape ‚Äî Function App (Flex Consumption / FC1)

Use **Flex Consumption (`sku_name = "FC1"`)**, NOT Premium EP1, NOT Y1 Consumption. FC1 supports SystemAssigned MI for runtime storage auth (the new pattern) and has the right cost shape for low-traffic editor workloads.

```hcl
resource "azurerm_function_app_flex_consumption" "app" {
  name                = var.function_app_name
  resource_group_name = data.azurerm_resource_group.swa.name
  location            = var.regions.primary
  service_plan_id     = azurerm_service_plan.functions.id

  storage_container_type      = "blobContainer"
  storage_container_endpoint  = "${azurerm_storage_account.func_runtime.primary_blob_endpoint}${azurerm_storage_container.deployments.name}"
  storage_authentication_type = "SystemAssignedIdentity"   # NOT a connection string

  runtime_name    = "python"      # or "node", "dotnet-isolated"
  runtime_version = var.runtime_version

  https_only                    = true
  public_network_access_enabled = true   # public + Entra-gated

  # NEVER set this. VNet integration on FC1 creates a `legionservicelink`
  # serviceAssociationLink that's a nightmare to remove ‚Äî TF can't clear it
  # cleanly, requires manual Owner-level `az functionapp vnet-integration
  # remove` to destroy the subnet. Don't go down this road.
  # virtual_network_subnet_id = ...

  identity {
    type = "SystemAssigned"
  }

  site_config {
    application_insights_connection_string = azurerm_application_insights.site.connection_string
    minimum_tls_version                    = "1.2"
  }

  app_settings = {
    # AzureWebJobsStorage via MI ‚Äî DO NOT also set the plain
    # AzureWebJobsStorage connection-string variant. Azure sometimes
    # injects it with an empty AccountKey and that overrides the MI
    # path with AuthenticationFailed.
    AzureWebJobsStorage__accountName = azurerm_storage_account.func_runtime.name
    AzureWebJobsStorage__credential  = "managedidentity"

    # Data-tier storage ‚Äî MI auth, no connection string
    DATA_STORAGE_TABLE_ENDPOINT = azurerm_storage_account.data.primary_table_endpoint
    DATA_STORAGE_ACCOUNT_NAME   = azurerm_storage_account.data.name

    # KV-referenced secrets (NEVER plain values)
    # SOME_API_KEY = "@Microsoft.KeyVault(SecretUri=${azurerm_key_vault_secret.some_key.versionless_id}/)"

    # Do NOT set:
    #   WEBSITE_VNET_ROUTE_ALL  ‚Äî there's no VNet to route through
  }

  auth_settings_v2 {
    auth_enabled           = true
    require_authentication = true
    unauthenticated_action = "Return401"   # NOT RedirectToLoginPage ‚Äî leaks intent
    excluded_paths         = ["/api/health"]   # narrowest possible

    azure_static_web_app_v2 {
      client_id = azurerm_static_web_app.primary.default_host_name
    }

    login {
      token_store_enabled = false   # REQUIRED ‚Äî auth_settings_v2 schema demands at least one login{} block
    }
  }
}

# FC1 host MI grants ‚Äî required or the host hangs in
# `FunctionsHostNotRunning` on cold start.
resource "azurerm_role_assignment" "func_runtime_blob" {
  scope                = azurerm_storage_account.func_runtime.id
  role_definition_name = "Storage Blob Data Contributor"
  principal_id         = azurerm_function_app_flex_consumption.app.identity[0].principal_id
}
resource "azurerm_role_assignment" "func_runtime_queue" {
  scope                = azurerm_storage_account.func_runtime.id
  role_definition_name = "Storage Queue Data Contributor"
  principal_id         = azurerm_function_app_flex_consumption.app.identity[0].principal_id
}
resource "azurerm_role_assignment" "func_runtime_table" {
  scope                = azurerm_storage_account.func_runtime.id
  role_definition_name = "Storage Table Data Contributor"
  principal_id         = azurerm_function_app_flex_consumption.app.identity[0].principal_id
}
# + Storage Table Data Contributor on the data SA for the Function App MI
# + Key Vault Secrets User on your KV (consider scoping per-secret rather than vault-wide)
```

### Storage Tables ‚Äî the giovanni gotcha

**Do NOT use `azurerm_storage_table` resources.** Background: the AzureRM provider's table resource uses the `giovanni` SDK for data-plane calls. Under AAD auth (which is the only auth we have since shared keys are disabled), the GetACL call returns an XML format giovanni can't parse ‚Äî every `terraform plan` fails refresh with:

```
Error: retrieving ACLs for Table "<name>" ... unexpected status 403
(Key based authentication is not permitted on this storage account.)
with expected element type <Error> but have <error>
```

HashiCorp closed this as **"Not Planned"** in [terraform-provider-azurerm#23142](https://github.com/hashicorp/terraform-provider-azurerm/issues/23142). The provider docs explicitly state: *"Shared Key authentication is always used for this resource, as AzureAD authentication is not supported when setting or retrieving ACLs for Tables."*

**Use one of these instead:**

1. **`azapi_resource`** with `Microsoft.Storage/storageAccounts/tableServices/tables@2023-05-01` ‚Äî control-plane API, AAD-native, properly TF-managed.
2. **`az storage table create --auth-mode login`** from a one-time bootstrap step in the deploy workflow. Less ideal (imperative, not declarative) but works for low-churn schemas.

Either way, declare the runner MI's `Storage Table Data Contributor` role grant in TF ‚Äî both approaches need it.

### RBAC propagation lag ‚Äî use `time_sleep`

Azure's RBAC backplane takes 30 seconds to 5 minutes to propagate new role assignments. Even with `depends_on` ordering correct, child resources that hit the data plane (containers, tables) can 403 because the grant hasn't propagated by the time TF tries to use it. Add a `time_sleep` resource between the runner role grants and the dependents:

```hcl
resource "time_sleep" "wait_for_runner_rbac" {
  depends_on = [
    azurerm_role_assignment.runner_data_blob,
    azurerm_role_assignment.runner_data_queue,
    azurerm_role_assignment.runner_data_table,
    azurerm_role_assignment.runner_runtime_blob,
    azurerm_role_assignment.runner_runtime_queue,
    azurerm_role_assignment.runner_runtime_table,
  ]
  create_duration = "120s"
}

resource "azurerm_storage_container" "deployments" {
  # ...
  depends_on = [time_sleep.wait_for_runner_rbac]
}
```

First apply takes ~2 extra minutes; subsequent applies no-op the sleep.

### Key Vault ‚Äî leave it at Allow

If you add KV-referenced secrets to your Function App, **keep the KV network ACL at `default_action = "Allow"` + `bypass = "AzureServices"`**. The SWA's KV-reference resolver (`@Microsoft.KeyVault(SecretUri=...)`) is **not** on the Azure trusted-services list. Flipping KV to `Deny` will silently break SWA secret resolution ‚Üí the SWA's Entra IDP loses its client secret ‚Üí site goes blank. `media-curation` learned this the hard way.

Also: when granting the Function App's MI access to KV, **scope per-secret** rather than `Key Vault Secrets User` on the whole vault. Audit finding from `media-curation`: vault-wide scope let the FA read the AAD app-reg client secret (highest-value secret in the vault) even though it didn't need it. Pattern:

```hcl
resource "azurerm_role_assignment" "func_kv_newsapi" {
  scope                = "${azurerm_key_vault.site.id}/secrets/newsapi-key"
  role_definition_name = "Key Vault Secrets User"
  principal_id         = azurerm_function_app_flex_consumption.app.identity[0].principal_id
}
# ... repeat per secret the function actually needs
```

### Pre-flight items that need CloudOps before your first apply

These can't be self-served; coordinate with the CloudOps team (currently Sai Gajavalli) **before** opening the Function App / Storage PR. Otherwise the apply 403s and you have to wait.

1. **Policy exemption for `default_action = "Allow"` on storage.** Tenant policy "Storage accounts should disable public network access" blocks our pattern unless an exemption is registered for the RG.
2. **Runner MI ABAC condition expansion.** The runner MI in your repo has `Role Based Access Control Administrator` on the per-site RG with an ABAC condition that allowlists specific role-definition GUIDs. By default it covers only `Key Vault Secrets User`. CloudOps needs to expand it to also cover the role-definition GUIDs for `Storage Blob Data Contributor`, `Storage Queue Data Contributor`, and `Storage Table Data Contributor`. Without this, the FA-MI role grants in TF will 403.
3. **Pre-grant runner MI data roles on the new SAs** (the chicken-and-egg from the bootstrap section above). The `time_sleep` covers the propagation lag but the initial grants still need to exist before the first refresh works.

CloudOps confirmed once per-repo ‚Äî they don't need to re-do this for every PR, only at infra bootstrap time for the site.

### Things you'll never need with this pattern

- VNet, subnets, NSGs, private endpoints, private DNS zones ‚Äî all gone from the pattern. If you find yourself reaching for any of these, stop and check with CloudOps first.
- `WEBSITE_VNET_ROUTE_ALL` setting on the Function App ‚Äî no VNet to route through.
- `infra/vnet.tf` ‚Äî don't create one.
- Storage account keys, SAS tokens, connection strings, `AccountKey=`, `SharedAccessSignature=` anywhere in code or app_settings. If you see one, that's a bug.
- A central private DNS zone link, `Ue1NePrdGHRunnerSubnet1`-style runner-subnet allowlists, IP firewalls ‚Äî that was the old pre-2026-05 pattern; the new model doesn't need them.

## What NOT to Do

- Do not add server-side code, APIs, databases, or any backend components **except Function Apps following the standard above**
- Do not add VMs, containers, AKS, or any IaaS resources
- Do not downgrade SWA to Free tier
- Do not modify the Terraform provider configuration EXCEPT to add `storage_use_azuread = true` (and the `azapi` / `time` providers) when introducing a Storage Account
- Do not modify the Terraform backend configuration
- Do not create additional WAF security policies
- Do not bypass Front Door by accessing SWA directly via the azurestaticapps.net URL
- Do not store secrets in code ‚Äî use GitHub repo secrets and KV references
- Do not modify `.github/CODEOWNERS` without understanding the review requirements
- Do not add npm, webpack, or any build tooling for the `site/` directory ‚Äî the deploy workflow does not run builds
- **Do not introduce a VNet, private endpoints, or private DNS zones.** The platform standard is public + Entra-only with MI for resource-to-resource. The `media-curation` app migrated AWAY from VNet-locked private endpoints in May 2026; that direction is locked.
- **Do not set `shared_access_key_enabled = true`** on any storage account you add. The Entra-only premise depends on keys being off.
- **Do not set `virtual_network_subnet_id`** on the Function App. FC1 leaves a `legionservicelink` serviceAssociationLink on the subnet that requires Owner-level manual cleanup to destroy. Never go there.
- **Do not use `azurerm_storage_table`** resources. Use `azapi_resource` or `az storage table create` ‚Äî see the giovanni gotcha section above.
- **Do not set the plain `AzureWebJobsStorage` connection-string app setting** on the Function App. Use `AzureWebJobsStorage__accountName` + `AzureWebJobsStorage__credential=managedidentity`. Azure platform sometimes injects the connection-string variant with an empty AccountKey and that overrides the MI path.
- **Do not flip KV `default_action` to `Deny`** ‚Äî SWA's KV-reference resolver isn't on the trusted-services list and will silently break.
- **Do not grant `Key Vault Secrets User` at the whole-vault scope** to the Function App MI ‚Äî scope per-secret instead.
- **Do not skip the `login {}` sub-block** in `auth_settings_v2` on the Function App ‚Äî the schema requires at least one even if it's just `token_store_enabled = false`. Plan fails with "Insufficient login blocks" otherwise.
- Do not add npm, webpack, or any build tooling ‚Äî the deploy workflow does not run builds
