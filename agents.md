## Limiting access to a specific user list (recommended for sensitive sites)

You can scope access at the **Entra layer** — Entra rejects sign-in for anyone
not on the list, before SWA ever sees the request. No SWA code changes, no
per-user invitations. Members are managed via a security group; add/remove
people from the group and access updates automatically.

**Two one-time setup steps** (need a tenant Application Administrator or
higher — ask `#techops` if you don't have it):

1. **Toggle "User assignment required" on the per-site SP.** This makes the
   app reg refuse tokens for unassigned users.
   ```bash
   az ad sp update --id 63bffde2-4bde-494f-b4f4-df1c019096a9 \
     --set appRoleAssignmentRequired=true
   ```

2. **Create an Entra security group, then assign it to the app.** Members of
   the group are the allow-list.
   ```bash
   # one-time: create the group (or reuse an existing one)
   az ad group create \
     --display-name "swa-automation-pw-users" \
     --mail-nickname "swa-automation-pw-users"

   # one-time: assign the group to the app (default app role)
   APP_SP_ID=$(az ad sp show --id 63bffde2-4bde-494f-b4f4-df1c019096a9 --query id -o tsv)
   GROUP_ID=$(az ad group show --group "swa-automation-pw-users" --query id -o tsv)
   az rest --method POST \
     --url "https://graph.microsoft.com/v1.0/groups/$GROUP_ID/appRoleAssignments" \
     --body '{
       "principalId": "'"$GROUP_ID"'",
       "resourceId": "'"$APP_SP_ID"'",
       "appRoleId": "00000000-0000-0000-0000-000000000000"
     }'
   ```

**Adding/removing users afterward:**
```bash
# add
az ad group member add --group "swa-automation-pw-users" --member-id <user-object-id>
# remove
az ad group member remove --group "swa-automation-pw-users" --member-id <user-object-id>
```

## Reverting to open access

```bash
az ad sp update --id 63bffde2-4bde-494f-b4f4-df1c019096a9 \
  --set appRoleAssignmentRequired=false
```

The group assignment can stay in place harmlessly — it just stops being
enforced.
